Why does pitch cancel to zero?

Circular cancellation explained with simple angle math (no quaternion algebra needed)

θ = real robot pitch (what we want to measure)
α = camera mount angle on robot (fixed, known from /tf_static)
θ+α = camera tilt in world (what Madgwick outputs)
0 = result of cancellation (bad!)
0Setup — what angles are involved
Imagine the robot is going uphill at 10°. The camera is bolted to the robot tilted 20° forward (mount angle).
Robot pitch in world    = θ = 10°   ← what EKF should estimate
Camera mount on robot  = α = 20°   ← fixed, from /tf_static in bag
Camera tilt in world   = θ+α = 30° ← camera sees 30° from horizon
1Madgwick filter output
Madgwick fuses gyro + accel from the D435i. The IMU is inside the camera, so it measures the camera's tilt in the world.
/camera/imu_filtered orientation = θ+α = 30° ← camera is 30° from horizon
Madgwick doesn't know about the robot. It only knows how much the camera is tilted.
2inertial_link_broadcaster (orientation_2d := false)
The broadcaster's job: figure out robot pitch from the camera's orientation. It looks up the static TF (α) and subtracts it:
robot_pitch = camera_tilt_in_world − camera_mount_angle
           = (θ+α) − α
           = θ = 10°   ← correct!
Then it broadcasts: base_link → inertial_link with pitch = θ
TF: base_link → inertial_link = pitch θ = 10°
So far so good. The TF tree now encodes the real robot pitch.
3EKF needs to use the IMU data — it must rotate it into base_link frame
The EKF receives /camera/imu_filtered. This message's frame_id is the camera gyro frame. Before the EKF can fuse it, it must transform the orientation into base_link frame.

It does this by looking up the TF chain:
base_link → inertial_link → camera_link → gyro_frame

Total rotation = θ (from broadcaster) + α (static mount) = θ+α = 30°
The EKF now applies this rotation to "undo" the sensor frame, expressing the IMU orientation in base_link:
orientation_in_base_link = IMU_measurement − TF_rotation
                       = (θ+α) − (θ+α)
                       = 0°
The EKF thinks the robot is perfectly flat.
No matter how steep the slope, it always sees 0° pitch — because the measurement and the transform were built from the same signal.
💡Everyday analogy
The shadow that follows you
Imagine you want to measure how tall you are by looking at your shadow on the ground.
But every time you move, your shadow moves with you — always the same length relative to you.
No matter how you tilt, your shadow always looks the same. You'd always think you're the same height.

The shadow (inertial_link pitch) was derived from you (IMU signal) — so comparing them tells you nothing.
The thermometer in a fridge you're trying to cool
You measure the fridge temperature. You cool it based on that reading.
But the thermometer is inside the cooling element — it always reads "cold" regardless of the actual fridge interior.

The sensor is part of the feedback loop it's trying to measure.

Why orientation_2d := true fixes Z

AWith orientation_2d := true
The broadcaster ignores the IMU signal and always publishes a flat transform:
TF: base_link → inertial_link = pitch 0°   ← identity, always
Now the EKF looks up the TF chain for the IMU:
base_link → inertial_link → camera_link → gyro_frame

Total rotation = 0° + α = 20° = α = 20°   ← just the static mount
EKF transforms IMU into base_link:
orientation_in_base_link = IMU_measurement − TF_rotation
                       = (θ+α) − α
                       = θ = 10°
EKF sees the real robot pitch → estimates Z correctly.
The TF rotation is now constant (α only), so it doesn't cancel the measurement.
But: the EKF's output pitch has smooth_lagged_data delay. When projecting the point cloud, the tilted base_link from the EKF is delayed → edge artifacts.

Why the robot IMU (/imu/data) solves both

BEKF uses /imu/data (direct chain, not through inertial_link)
If the robot's own IMU (/imu/data) connects to base_link via a direct static TF:
base_link → imu_link   ← pure static TF, does NOT pass through inertial_link
EKF transform for /imu/data:
Total rotation = β (imu_link mount, fixed)   ← constant, no circular path
EKF sees:
orientation_in_base_link = IMU_measurement − β = θ   ← real pitch!
Meanwhile, inertial_link_broadcaster still runs with orientation_2d:=false using D435i:
base_link → inertial_link = θ   ← real, responsive, no lag
Point cloud projection:
depth_frame → inertial_link (real θ, instant) → base_link → odom
Both problems solved:
✓ Z correct — EKF uses robot IMU via direct static chain (no cancellation)
✓ Sharp map edges — D435i pitch in inertial_link is real-time (no EKF delay)
Prerequisite: check that /imu/data frame_id has a direct base_link → imu_link static TF in the bag, not routed through inertial_link.
